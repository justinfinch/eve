Hardware manual · for self · 2026-06-22

Build It, Understand It

A bench manual for assembling the Mini-Molecule (Concept A) hardware — and, more importantly, for understanding every part well enough to explain it.

Audience. You, the hands-on builder. You write the software; this is the hardware half.

The ask. Build rung-by-rung; for each rung, write the "what's happening in the silicon" note. Success = you can explain it, not that the demo passes.

The one rule. When it breaks, suspect the physical layer first.

0Orientation

You are building Concept A — the Mini-Molecule + Cloud Workbench: a small board that self-identifies on a bus and is driven from a browser, reading a sensor and driving an actuator, end to end. You will write the software with skills you already have. This manual is the other half: the physical build, and the understanding.

The answer, stated first

The deliverable is hardware literacy, not a working rig. A live bus you can't explain is worth less than a humbler one you deeply understand. So every rung ends the same way: write down what's actually happening in the silicon. That write-up is the deliverable — the repo is a textbook you author by building.

And the corollary that governs all debugging: when something breaks, suspect the physical layer first. In hardware, physical failures masquerade as software bugs — a crystal mismatch reads as a "CAN bug," static damage reads as "flaky firmware," a cold contact reads as a "code bug." The whole manual is built to make the physical layer observable so you stop blaming code.

1The mental model

Three mental models you carry as a software dev are silently false in hardware. Internalize these before you wire anything.

Trap 1
Errors are catchable.
Hardware has no try/catch. A wrong value is current going where it shouldn't — physical, permanent. No git reset. Measure before you connect.
Trap 2
It compiled, so it's right.
Clean firmware still returns garbage from a cold contact, wrong SPI mode (CPOL/CPHA), a byte-order flip, an unconnected MISO. "Code is correct" → go to the physical layer.
Trap 3
State lives in variables.
A floating GPIO reads noise — no default 0. Config registers persist inside the chip across re-flashes. Caps and rails carry state. The twin of "initialize your variables": pull resistors, register config, decoupling.
What transfers: not every instinct betrays you — "fail fast on the riskiest assumption" is correct, and it's exactly why CAN moves early (rung 2).

The meta-skill — misattribution. The single deepest pattern: physical failures wear software costumes. Learning to suspect the physical layer first is the literacy this whole POC is training.

3Day one — the arrival → bring-up ritual

There is no package-lock.json for atoms — no hash guarantees you got the right bytes. Every part is guilty until proven innocent: clones, mislabels, wrong crystals, DOA, missing SMD. Treat the box landing as ready to audit, not ready to build.

  • 1 · Audit before assembly. ESD strap on first. Decode chip top-markings vs the datasheet (ADS1220 is ADS1220; Pico is genuine RP2350; transceiver is TJA1050). Read both MCP2515 crystal cans and tape-label them — a 30-second step that pre-empts the entire "CAN bug" spiral. Visual-check pins, bridges, cold joints.
  • 2 · Smoke-test each part in isolation. Power it alone, watch the power meter, feel/smell for heat. Verify against math, not an instrument you also don't trust. Use the chips' own self-tests (MCP2515 LOOPBACK; register write-then-read-back). Passive parts test with a meter only.
  • 3 · Integrate one variable at a time — so any failure is unambiguously the one thing you just added. Binary search, applied to atoms.
flowchart TD A([Box arrives]) --> B{ESD strap on?} B -->|no| B1[Put it on first] B1 --> C B -->|yes| C[AUDIT: decode chip markings vs datasheet] C --> D[Read both MCP2515 crystal cans, tape-label] D --> E[Visual: bent pins / bridges / cold joints] E --> F[SMOKE TEST each part alone] F --> G{Power meter} G -->|hot / shorted| G1[Cut power: chip backwards or dead] G -->|normal| H[Self-test: LOOPBACK / register read-back / math check] H --> I{Passes in isolation?} I -->|no| I1[Fix this part before it contaminates a multi-part debug] I -->|yes| J[INTEGRATE one variable at a time] J --> K([Any failure = the one thing you just added])

The receiving-inspection discipline. The box is ready to audit, not ready to build.

RUNG 0Power, external LED, multimeter

New concept: GPIO, current-limit resistor, 3.3V logic, the meter as debugger. Safest first contact.

Pico GP15 ──[ 330Ω ]──>|── GND     (anode > resistor > GP15; cathode > GND)
                        LED

What's happening in the silicon: GP15 is a push-pull output — driven high it sources ~3.3V; the 330Ω limits current to ~(3.3−1.8)/330 ≈ 4.5 mA so neither the LED nor the GPIO pad burns. The onboard LED won't work here (it's behind the CYW43 chip on the W) — which is why you use an external one.

Verify: measure GP15 toggling 0V↔3.3V with the multimeter before connecting; measure across the resistor for current direction.

Write-up: why a current-limit resistor exists and what happens without it.

RUNG 1Browser↔device loop + self-ID [gpio]

New concept: serialization, the host↔device boundary, capability announcement. A software dev's home turf — build the instrument first.

Wiring: none beyond rung 0 — uses the Pico's native USB. Browser → Web Serial API → Pico directly (Chrome-only), the browser-to-serial pattern John Lauer pioneered.

What's happening: on power-up the firmware announces {id, name, capabilities:["gpio"]} over USB-CDC serial; the browser parses it and renders an LED control. This is the molecule's self-ID spine — every later rung appends a capability.

Verify: toggle the LED from the browser; confirm the capability string round-trips.

Write-up: where the host↔device boundary sits and how a capability list makes the board plug-and-play.

RUNG 22-node CAN — the big one

New concept: bus arbitration, differential signaling, bit-timing. "Bloody me early" on the highest-gotcha, Adom-signature tech. Two gotchas straight from the BOM build note:

⚠ Gotcha (a) — Crystal frequency. Cheap MCP2515 modules ship 8 MHz or 16 MHz, by batch. The firmware bit-timing config must match the actual crystal or the bus never syncs. You already read & labelled the cans in §3 — set the firmware constant to match. A mismatch reads as a "CAN bug" for hours — the #1 misattribution trap.
⚠ Gotcha (b) — the 5V/3.3V MISO seam. The TJA1050 wants 5V; the Pico is 3.3V. SPI is 4-wire — the Pico drives SCK/MOSI/CS (safe), but MISO is driven by the module, so a 5V-powered module can push 5V into a Pico GPIO and damage it over time. Simplest-first: run the module at 3.3V (fine on a short bench bus); or put the BSS138 on MISO; or board-mod to feed only the TJA1050 5V.

Wiring — SPI from Pico to its MCP2515 (⚙ derived):

Pico            MCP2515 module
GP18 (SCK)  ─── SCK
GP19 (MOSI) ─── SI
GP16 (MISO) ─── SO     <── route through BSS138 if module is 5V-powered
GP17 (CS)   ─── CS
GP21 (INT)  ─── INT
3V3 / 5V    ─── VCC    (see seam note above)
GND         ─── GND

The CAN bus between the two nodes — differential pair + termination at both ends, rendered below.

Why you can't see CAN with a single-ended probe — and why the logic analyzer goes on the SPI lines, not the bus.

What's happening in the silicon: CAN is a differential bus — CANH/CANL swing in opposite directions, so common-mode noise cancels; that's why it's industrial-robust and why a single-ended probe shows you nothing useful. 120Ω terminators at each end stop reflections. The MCP2515 is an SPI-controlled CAN controller; the TJA1050 is the transceiver that turns its TX/RX into the differential pair. Config registers live inside the MCP2515 and survive a re-flash (Trap 3).

De-risk before two nodes talk: prove one node alone in LOOPBACK mode (TX→RX internal — no bus, no second node). Then bring the bus up. If it won't sync, swap the Golden Reference Node in for your hand-built node to bisect: wiring+firmware vs the bus itself. Put the logic analyzer on the SPI lines — confirm CS asserts and bytes move.

Verify: LOOPBACK passes → register read-back matches → two nodes ACK → a frame from A appears on B.

Write-up: what differential signaling buys you, why termination matters, and the crystal/MISO seam in your own words. Reach here understanding both gotchas and you can speak to CAN in an interview — the whole point.

RUNG 3Potentiometer via the built-in ADC

New concept: analog↔digital, resolution, reference voltage. Validates the whole software loop at ~zero hardware risk.

3V3 ──[ pot top ]
        wiper ─── GP26 (ADC0)
GND ──[ pot bottom ]

What's happening: the pot is a voltage divider; the wiper feeds 0–3.3V to GP26 (ADC0). The RP2350's 12-bit SAR ADC returns 0–4095 ≈ (Vin/3.3) × 4095. This rung is a variable-isolator: it proves the entire browser→device→measure→stream loop with the built-in ADC, so when you swap to the ADS1220 over SPI (rung 4), the SPI chip is the only new variable.

Verify: rotate the pot, watch the value stream; endpoints read ≈0 and ≈4095.

Write-up: what "12-bit resolution against a 3.3V ref" means in volts-per-count.

RUNG 4Same pot via ADS1220 over SPI

New concept: SPI (CPOL/CPHA), register config, 24-bit delta-sigma, PGA. Only the SPI chip is new → failures are unambiguous.

Pico            ADS1220        (shares the SPI bus with MCP2515, separate CS)
GP18 (SCK)  ─── SCLK
GP19 (MOSI) ─── DIN
GP16 (MISO) ─── DOUT
GP20 (CS)   ─── CS
GP14 (DRDY) ─── DRDY
3V3         ─── AVDD/DVDD
GND         ─── GND
pot wiper   ─── AIN0   (AIN1 ─── GND)

What's happening: the ADS1220 is a 24-bit delta-sigma ADC — it oversamples and noise-shapes, trading speed for resolution, which is why it beats the built-in SAR for precision. Set the right SPI mode (CPOL/CPHA) or every byte is garbage despite clean firmware (Trap 2). Its config registers persist in-chip (Trap 3). It has a PGA and onboard IDAC current sources for ratiometric sensor reads.

Verify against math: short AIN0–AIN1 → read ≈0; known divider → code must equal (Vin/Vref) × 2²³ by hand. Logic analyzer on SPI if a byte looks wrong.

Write-up: SPI mode, what delta-sigma buys over SAR, why you trust the number.

RUNG 5PWM a motor via transistor + flyback

New concept: PWM, current, transistors, power rails, back-EMF. Actuation + power discipline.

                    +5V rail (power module, NOT Pico 3V3)
                       │
                   [ motor ]
                       │
          1N4007 ►|───┤   (flyback across motor, cathode to +5V)
                       │
Pico GP13 ──[ 1kΩ ]── base │ collector
                   2N2222 (NPN)
                        emitter ── GND ── (common GND with Pico)

What's happening: PWM is a fast on/off square wave; duty cycle sets average power, so the motor sees an effective fraction of 5V. The 2N2222 switches the motor current the GPIO can't source. The flyback diode clamps the inductive back-EMF spike when the motor switches off — without it that spike kills the transistor (Trap 1: current going where it shouldn't, permanent). Power comes from the 5V rail with a common ground so the switch reference is shared. A motor stall can brown-out a shared rail (Trap 3) → keep its supply separate from the Pico's.

Verify: sweep duty 0→100% from the browser; confirm diode orientation with the meter before powering.

Write-up: why the transistor and flyback exist; what duty cycle maps to physically.

RUNG 6AI layer + shareable permalink

New concept: none hardware — pure software, layered last. NL → test-plan → run on the molecule → plain-English readout → permalink for reproducibility. It sits cleanly on top because every rung below already exposes a clean capability over self-ID.

5The diagnosis playbook

When a rung misbehaves, run the misattribution check before opening the firmware. The default suspicion is the physical layer.

flowchart TD S([Symptom: it does not work]) --> P{Power meter: current sane?} P -->|near-zero| P1[Nothing running: check power, contacts, CS asserting] P -->|too high + hot| P2[Cut power: chip backwards / shorted / dead] P -->|normal| Q{Did it EVER work?} Q -->|worked, now flaky| R[Suspect ESD / cold breadboard contact / loose dupont] Q -->|never worked| T{Single part passes self-test in isolation?} T -->|no| T1[Bad / clone / DOA part: bisect with Golden Reference Node] T -->|yes| U{Logic analyzer: do the bits move?} U -->|no bits / CS not asserting| U1[Wiring / pin map / SPI mode CPOL-CPHA] U -->|bits move, value wrong| V[Crystal mismatch / byte-order / config register] V --> W[ONLY NOW suspect firmware logic]

The order is the lesson: power → history → isolation → wire-level → then code. Your instinct starts at W; literacy starts at P.

The ask

Build Concept A one rung at a time, validating each in isolation before stacking the next. For every rung, write the short "what's actually happening in the silicon" note — physics, why-this-chip, failure modes, code↔silicon mapping. That trail is the deliverable. Success = being able to explain the hardware, not making the demo pass. And when it breaks: suspect the physical layer first.
